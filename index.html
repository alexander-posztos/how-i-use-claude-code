How I Use Claude Code

EnglishDeutsch

TL;DR

  • Context is the metric that matters. Keep each session under the ~200k window.
  • Maintain your CLAUDE.md actively; keep it short. Only add rules for recurring or annoying things.
  • Close the loop. Let the agent prove that the code works in more than one way.
  • Settle the plan before you build. Have it grill you on the decisions.
  • Use voice mode. Dictating long, detailed prompts is far faster than typing them.

1. Why This Doc

I've been using Claude Code for about half a year now, and I've gotten pretty effective with it. To share what I've learned, I went back through my own sessions and analyzed them with Claude Code to find the patterns that have a big impact on how I work on a daily basis: about 15,000 prompts over 6 months across 20 project folders, plus the full session transcripts. The quotes and prompts you see in here are actual ones from me, so they'll have a lot of typos.

2. The Mental Model

The main thing I always try to remember is that the agent is a brilliant reasoner with no memory of its own. Every session begins blank, and the context window is the whole world it can see. Claude Code's main job, other than running commands on your machine, is to gather the right context and send it to the model.

So one question sits under nearly every habit in this doc: how do I get every relevant piece of information in front of it without blowing up the token budget? CLAUDE.md, skills, status.md files, handoff briefs are all answers to that one question.

The most basic answer is structural: every problem becomes a project folder. When I start a new topic or project, I begin by creating a new directory, dropping the relevant files in (or telling Claude to go find them in Downloads), writing a CLAUDE.md, and getting to work. For example, my income tax return is a "project" with its own folder, notes, and CLAUDE.md.

3. Starter Pack

If I were starting from scratch on a new Claude Code setup, I'd do it like this:

  1. Write a CLAUDE.md first. Before anything else, in your main repo: the general stack (and why you chose it), how to run and test things, and a few hard rules - mine include "never use em-dashes". Keep it very concise, under a screen; it grows naturally as you keep working (section 5).
  2. Understand a few important commands - they matter more than prompting tricks:
    CommandWhatMy usage
    /clearFresh sessionConstantly - one task, one session
    /rewindRestore a checkpoint (chat + files)Undo-and-reprompt instead of arguing
    /resumeReopen a previous sessionRe-entry after a break
    /contextShow what's filling your windowRun it early - see section 6

    /model, /usageModel switch, cost checkDownshift for mechanical work
    @file, !cmdReference files, run shell inlineAnchor prompts to real artifacts
    /compactSummarize context in placeAlmost never - handoff + clear instead
  3. Build habits that keep context clean and short:
    • one task per session
    • the model always knows how to verify its changes
    • planning or /grill-me for anything non-trivial
    • the handoff skill to carry work into a fresh session (section 7)
  4. Add skills. Copy mine or Matt Pocock's - they drop into ~/.claude/skills/ - and make a habit of using them. Skills are mostly just markdown files, so change them however you want: my to-prd creates HTML instead of the original's markdown (section 9).

4. Setup & Config

You can configure Claude Code to your liking via ~/.claude/settings.json - and the easiest way to do that is, of course, to ask Claude Code to apply the changes itself. What I've changed, and why:

  • My default model is Opus, with the 200k window - the 1M variant deliberately disabled. Two reasons. First, the smaller-context models anecdotally have better base performance. Second, the smaller window forces you to be careful with your context: you scope tasks smaller and clear more often, which keeps you in the smart zone (section 6) by default. A bigger window mostly buys more room to get dumb in. Since Fable 5 came out a few days ago, I am currently using that instead, but it will probably not stay my daily driver due to price. If you can, use Fable though, as it is much more capable.
  • Effort max + always thinking (both configured in settings.json). I'd rather have the model one-shot a prompt than save on reasoning and risk having to re-prompt - and with thinking always on, I never need magic words like "think hard". There's also a new mode called ultracode: very token-hungry, and it does more than just make the model think harder - it fans the task out to parallel subagents (read more here). Use it when you have tokens to burn and a problem worth pointing them at.
  • Attribution off. Commits don't say "Generated with Claude Code" - I review every diff before it lands, so the commit is mine in the sense that matters.
  • Use voice mode for prompting. Activate it with /voice; once it's on, you just hold space and talk into the input. Really nice for describing an idea thoroughly, and it doubles as thinking out loud. It understands context-dependent things like library names - say "context seven" and Claude knows you mean context7, so transcription errors don't need fixing.
  • Custom statusline showing the project I'm in, the model in use, and a bar for how much of the context is used up. Really useful for seeing how close you are to the dumb zone, and when to get ready to hand off to a new session.
  • Plugins: the official Claude Code plugin marketplace offers lots of them. Honestly they're not that essential, since most can be replaced by plain skills, but they're maintained and easy to install. Some I use: pyright-lsp, frontend-design, posthog (toggled on when needed), academic-research-skills for thesis work. MCP servers I keep mostly disabled - why, in section 9.
  • Random stuff:
    • I have auto-update enabled (Claude Code ships new features constantly, you want them fresh).
    • You can set color themes via /config.
    • A usage trick: the limiting part of your Claude Code subscription is usually the five-hour usage window, and it doesn't reset every five hours on its own - it only starts once you actually use it (a bit annoying, but you can work around it). So I have a daily cron job at 6 AM that fires one single prompt just to start the window: when I wake up around 8 or 9, it resets in just two and a half hours instead of five.
    • I run Claude in --dangerously-skip-permissions mode, aliased to yolo. Do this once you feel confident it won't delete your whole PC.

The script (with a short README) is in my dotfiles.

5. CLAUDE.md & Memory

CLAUDE.md is a markdown file in the repo root that gets loaded into every session. It's where you put the project's standing instructions, so you don't have to explain the basics again every time. Mine has:

  • the stack
  • eight numbered hard rules
  • domain gotchas (Austrian invoicing law)
  • how to run commands and tests, plus the test accounts
  • what's off-limits: the prod DB is read-only, I handle git myself, never start the dev server

Two things make it actually work:

  • Add rules, not documentation. Your CLAUDE.md and README.md should be two different things. CLAUDE.md should be short, and made to prevent wrong actions by your agent. For example, "prefer early returns instead of nested ifs."
  • Iterate on it whenever you are annoyed with your agent, e.g. if it keeps making the same mistake and you have to correct it, say "never make this mistake again, add a concise sentence in the CLAUDE.md file."

Claude Code recently started to keep auto-memory per project - small fact files it writes itself between sessions (decisions, feedback, gotchas). I'm not really a fan: the memories are not actively maintained by you, so they can contain stale or confusing info. You can deactivate it with "autoMemoryEnabled": false in settings.json. For long-running projects I add a status.md instead: CLAUDE.md is the stable reference, status.md is the dynamic "where are we" that every re-entry session reads first.

6. Context & the Dumb Zone

My three most-used commands, by actual count: /clear (570), /rewind (484), /resume (419). These commands are there for session management, which is one of the most important skills in using Claude Code: managing context matters more than writing prompts. /resume simply reopens an earlier session when I want to pick an old task back up - the other two run through the next section.

There is a reason I use /clear so often and maintain short context sizes. The idea comes from Matt Pocock: he argues that a model is smart at small context sizes (the smart zone) and gets worse and worse as more context is added. Remember that for every prompt, the whole history of the chat is the prompt. He puts the boundary at around 100-150k tokens. After that you get into the dumb zone, where models seem to get sloppier, more forgetful and confidently wrong. You can use long context for retrieval, for example, if you have a lot of files and want to search something specific in them. But for most coding tasks, keep a short context window.

Use /context to get a better understanding of how your context window is used up: it shows you exactly how many tokens are used, e.g. by your CLAUDE.md, MCP servers and so on.

7. The Session Loop

  1. Start a clean context. If you are already in a session, use /clear.
  2. Investigate first. In this first session you just try to find the context, use web search and so on. The result is an artifact: a handoff prompt, a PRD file or a GitHub issue (section 8).
  3. Restart clean, one anchored ask, verification included - a fresh session anchored to that artifact:
    work on github issue 26, make sure to test it e2e on local with email sending to console and playwirght
  4. Let it run. Claude does bursts of 100-500 tool calls in one go - let it work. When I see it going in the wrong direction or doing something not as I wanted it to, I just press escape to stop it. And then, usually, a one-liner steering it back in the right direction is enough.
  5. Prove it works. Have Claude verify the change in the real browser, run the tests and check the logs - more on this in section 10.
  6. Review and commit myself. I go through the diff and commit on my own - Claude's last message typically ends with "changes are uncommitted, git is yours".
  7. Hand off and start over. Once the task is done, I run /handoff, then /clear - and the loop starts again.

Handoff + clear instead of compacting

The default behavior of Claude Code when the context fills up is a compaction: it summarizes the conversation in place. I don't like compaction, since you do not have control of how / what happens, and I almost never let it: 28 compacts vs 570 clears in six months. Instead I use my /handoff skill, which gives me a prompt I can still edit if I want before pasting it into the next fresh session - context, what's done, next goal, "files to read first, in order", and a working agreement.

Rewind instead of arguing

If you realize that you're going down a wrong path, you can undo the conversation and / or the code and /rewind to a previous point in the conversation. This is useful as it saves context: instead of debating a failed attempt back to health, I rewind and re-prompt with what it taught me. It also makes it cheap to just try out different approaches.

8. Planning: Grill, Then Build

The main reason why you get bad results isn't usually the model performance - rather, it's that the model and you have different images in mind of how the solution should work. The /grill-me skill will ask you tons of questions, just to make sure that you are both on the same wavelength.

During the "grilling process" the model will give you a few options + its recommendation for each question. Usually this will surface a lot of problems or ideas that you haven't really thought about - it's kind of like rubber ducking as well (rubber ducking used to be a thing pre-AI, but not sure if anyone still does it).

  1. Negotiate with /grill-me.
  2. Freeze the result as a PRD with /to-prd - a skill that turns the negotiated context into a self-contained HTML spec. The PRD is a living document: my biggest migration ran 18 implementation sessions off one PRD.
  3. Execute in fresh sessions.

For multi-week efforts the handoff brief carries a block protocol ("pi" is my second-model reviewer, section 11):

After each block: tests green locally, commit, run pi per the loop doc, address every URGENT, iterate until pi returns zero. Then a one-paragraph summary, then wait for my sign-off before block N+1.

I use the /grill-me pattern for bigger changes. Most sessions usually start from a GitHub issue I filed earlier.

9. Skills (and MCP)

A skill is just a folder with a SKILL.md in ~/.claude/skills/ (global) or <repo>/.claude/skills/ (per project). The folder contains the instructions and optionally bash scripts or other assets. You can use a skill by typing /name, or set the description in its frontmatter so it can be triggered by Claude itself.

I mainly think of a skill as a prompt that I got right, that did exactly what I wanted. If you notice yourself typing the same prompt again and again, create a skill for that. The easiest way is to actually use the skill-creator skill (skill-ception). It takes almost nothing - this is my most-loved skill, grill-me, in its entirety:

---
name: grill-me
description: Interview the user relentlessly about a plan or design
  until reaching shared understanding. Use when user wants to
  stress-test a plan or mentions "grill me".
---

Interview me relentlessly about every aspect of this plan until
we reach a shared understanding. Walk down each branch of the design
tree resolving dependencies between decisions one by one.

If a question can be answered by exploring the codebase, explore
the codebase instead.

For each question, provide your recommended answer.

Skills vs MCP

There's mainly two ways to give Claude more capabilities, Skills and MCP. It can be confusing what the difference is. Functionally you can think of them doing the same thing for you. The important part is about the context though:

  • An MCP server used to inject all of its tool schemas into context at session start, used or not - a typical five-server setup ran ~55,000 tokens before you'd typed a word, a quarter of the window gone. Claude Code now defers most MCP schemas and loads them on demand (tool search), which fixes the worst of it - but every connected server still adds baseline context, and the schemas do load once they're needed. /context makes this visible.
  • A skill is progressively disclosed: at startup only its name and one-line description load; the full instructions are read when invoked, supporting files only as needed.

So my rule: if a good CLI exists, wrap it in a skill instead of connecting a server. Usually a public skill already exists that you can install, or you can point Claude Code to the CLI and just tell it to create / write the skill for you.

10. Verification

To get usable results from a one-shot prompt, you need to give your agent ways to confirm, test and verify their implementation. The more resources you give it, the better. This is of course highly dependent on your project, but for web dev you can have linting, browser testing, logs and so on.

The rule is confirm it more than one way. Each mode catches what the others are blind to - a passing unit test says nothing about whether the button actually renders. For bigger changes, I usually tell Claude Code to give me a manual checklist as well, as some problems are not obvious to the agents.

For a bug I want the loop closed end to end, in this order:

  1. Reproduce as a failing test first. Investigate, then write a test that fails on the bug - red before any fix, so I know the test actually bites.
  2. Implement the fix.
  3. Prove it two ways: exercise the fix in the browser with playwright, then rerun the red test and watch it go green.
  4. Lock it in: add that test to the suite as a regression guard, then run the whole suite to confirm the fix broke nothing else.
  5. Hand me the manual check - the click path, the URL, what I should see - so I can confirm it the last way a machine can't.

I've wrapped this exact loop into a skill - /close-the-loop - so I can trigger the whole sequence by name instead of re-explaining it every time. It lives in my public dotfiles repo (section 12).

Git stays mine

I don't hand off committing and pushing to the agents, just because I want to understand and review the code myself. This is not really necessary from a technical standpoint - you can definitely let the agent take over. I just like to really understand how the agent implements stuff. Also, sometimes I catch the agent implementing stuff way too complicated and notice that my prompt wasn't specific enough - so that's also a benefit of handling the git stuff yourself.

11. Reviews & Subagents

A pattern that works really well for me is the implementer / reviewer pattern: the agent that implemented something should not be the one reviewing it. The reviewer runs in a fresh context, so it doesn't know what the implementer was thinking and just judges what is actually there. If the model missed an edge case while writing the code, it will most likely miss it again when reviewing its own work.

I take this one step further and use a different vendor's model for the review: my /pi-review skill spawns the pi coding agent running OpenAI's GPT-5.5, which reviews the diff against my CLAUDE.md conventions. A second model has different blind spots, so it catches things the model that wrote the code just doesn't see. I go through the findings one by one ("no thats ok, keep that. any other issues?"), and during big migrations this runs as a loop: review every block, fix every URGENT, iterate until clean.

I rarely use subagents. When I do, it's mainly for broad searches, research or audits. Sometimes I also use them when I want to use up my tokens - here's an example where I just wanted it to find a bug without my help:

USE ALL YOUR SKILLS AND 5 SUBAGENTS TO FIND THE PROBLEm

(It found the problem.)

12. Further Work

My custom skills (grill-me, handoff, to-prd, close-the-loop, pi-review) live in my public dotfiles repo: github.com/alexander-posztos/dotfiles. The setup itself is never finished - what I want to change next:

  • Replace grill-me with grill-with-docs (Matt Pocock). Same relentless interview, but it first reads a CONTEXT.md glossary and the repo's decision records, challenges fuzzy terminology against them, and writes resolved decisions back into the docs - so every grilling session leaves standing context behind for all later sessions. Needs a one-time CONTEXT.md per repo; grill-me stays for projects without a codebase.
  • Move the hard rules from CLAUDE.md into hooks - scripts that run deterministically on agent events, like before every tool call. A PreToolUse hook can block writes to my prod database even in yolo mode, and a PostToolUse check can reject an em-dash the moment it's written. CLAUDE.md rules are followed most of the time; hooks fire every time.
  • Let Claude work the issue board via the GitHub Action. Most of my sessions already start from an issue I filed earlier (section 8); with the action installed, an @claude mention on the issue implements it and opens a pull request headlessly. Review stays mine either way.

Agentic coding, vibe coding, or whatever you want to call it - this changes constantly, and I try to keep this doc up to date as best I can. In the end, I think the most important thing is to just work with the tools yourself a bit and get a feel for them. You can use this guide as a starting point / inspo - but your workflow will for sure look totally different. Have fun!
